# ⚡ EIP-3009: Transfer With Authorization

### The Gasless Signing Standard Powering Agentic Payments


***

## 🔍 What Is EIP-3009?

**EIP-3009** is an Ethereum token standard that lets a token holder authorize a transfer using just a **cryptographic signature — no on-chain transaction, no gas required from the sender**. It introduces two new functions to any ERC-20-compatible token contract: `transferWithAuthorization` and `receiveWithAuthorization`.[^1][^2]

The standard was proposed in 2020 and is natively implemented by **USDC** and **EURC** (Circle) and **AUSD** (Agora) — the primary stablecoins used in agentic payment systems today.[^3]

> 🏷️ **Related protocols \& standards:** `#ERC-20` `#EIP-712` `#EIP-2612` `#Permit2` `#x402` `#USDC` `#AgentCash` `#AP2`

***

## 🧠 Core Concept: What Problem Does It Solve?

In standard ERC-20 token transfers, **the sender must initiate and pay for the on-chain transaction** using the native gas token (ETH, MATIC, etc.). This creates two critical problems for autonomous systems:

- 🔴 **Gas dependency** — the agent wallet must always hold native ETH just to move stablecoins
- 🔴 **Sequential nonces** — transactions must be ordered and executed one-by-one, blocking parallel operations

EIP-3009 eliminates both by moving the **authorization step off-chain** and delegating the gas payment to a third-party relayer (the *facilitator*).[^4][^3]

***

## ⚙️ How EIP-3009 Works (Step-by-Step)

### Step 1 — 🖊️ Off-Chain Signing (The Agent / User)

The token holder (or AI agent wallet) never submits a blockchain transaction. Instead, it creates an **EIP-712 typed structured message** containing these exact fields:[^1]

```
TransferWithAuthorization {
  from:        address   // payer's wallet
  to:          address   // recipient's wallet
  value:       uint256   // token amount
  validAfter:  uint256   // unix timestamp — earliest valid time
  validBefore: uint256   // unix timestamp — expiry deadline
  nonce:       bytes32   // random 32-byte value (NOT sequential)
}
```

The agent signs this message with its private key → producing a `(v, r, s)` ECDSA signature. **Nothing is sent to the blockchain yet**.[^4][^1]

***

### Step 2 — 📦 Signature Delivery (To the Facilitator)

The signed message is transmitted to a **relayer or facilitator** — an off-chain service responsible for gas payment and on-chain submission. In the x402 ecosystem, this is the `x402.org` facilitator. In agentic wallets, this is the wallet infrastructure provider (e.g., AgentCash, Cobo).[^5][^3]

***

### Step 3 — ✅ On-Chain Execution (The Contract)

The facilitator calls `transferWithAuthorization(from, to, value, validAfter, validBefore, nonce, v, r, s)` on the token contract. The contract:[^1]

1. **Verifies** the EIP-712 signature matches the `from` address
2. **Checks** the nonce has not been used before (`authorizationState`)
3. **Validates** the current time falls within `validAfter` and `validBefore`
4. **Executes** the token transfer atomically
5. **Marks** the nonce as consumed — preventing replay[^4][^1]

The facilitator pays all gas. The agent's wallet only ever needed to hold the **stablecoin being transferred**, nothing else.[^5]

***

## 🔑 The Random Nonce: A Critical Design Choice

Unlike EIP-2612 (Permit) which uses **sequential contract nonces**, EIP-3009 uses **random 32-byte nonces**. This seemingly small detail has massive implications:[^6][^3]


| Feature | EIP-3009 | EIP-2612 Permit | Permit2 |
| :-- | :-- | :-- | :-- |
| **Nonce type** | Random 32-byte | Sequential (contract) | Flexible |
| **Parallel signing** | ✅ Yes | ❌ No | Partial |
| **Pre-approval step** | ❌ Not needed | ✅ Required | Required |
| **Token support** | USDC, EURC, AUSD | Mainstream ERC-20s | Universal |
| **Best for** | Agent micropayments | DeFi allowances | General DeFi |
| **Replay protection** | Per-nonce state | Sequential ordering | Bitmap |

Because nonces are random, **an agent can sign 100 micro-payments simultaneously** without ordering conflicts — essential for autonomous agents running parallel workflows.[^3]

***

## 🤖 How EIP-3009 Powers Agentic Payments

This is where EIP-3009 transitions from a wallet convenience feature to **foundational infrastructure for the agent economy**.

### 🔗 The x402 Protocol Integration

The x402 payment protocol — the open standard on which AgentCash is built — uses EIP-3009 as its **core on-chain settlement mechanism**. The full agentic payment flow looks like this:[^7][^3]

```
AI Agent
  │
  ├─► 1. HTTP GET /paid-resource
  │
  ◄── 2. HTTP 402 Payment Required
  │       { token: USDC, amount: 0.01, to: 0xABC..., deadline: +30s }
  │
  ├─► 3. AgentCash signs EIP-3009 authorization (off-chain, no gas)
  │
  ├─► 4. HTTP GET /paid-resource
  │       Header: payment-response: { signed EIP-712 message }
  │
  [Facilitator submits transferWithAuthorization on-chain, pays gas]
  │
  ◄── 5. HTTP 200 OK + resource delivered
```

The agent **never pays gas**, **never holds ETH**, and **never blocks on nonce ordering**.[^8][^5]

***

### 🧩 Why This Fits the Agentic Model Perfectly

Autonomous agents have fundamentally different needs from human wallet users:[^5]

- 🕐 **Speed** — agents need to pay in milliseconds, not wait for manual approval or nonce resolution
- 🔀 **Parallelism** — agents may run dozens of tasks simultaneously; random nonces allow concurrent payment signing
- 🔐 **No ETH management** — agents should only hold the payment asset (USDC), not manage a gas float
- ⏱️ **Time-bounded authority** — `validBefore` expiry timestamps mean each signed authorization auto-expires, limiting blast radius if a signature is intercepted
- 🚫 **Replay prevention** — consume-once nonces ensure a captured signature cannot be reused by a malicious actor[^9][^4]

***

## ⚠️ Limitations to Know

EIP-3009 is not universally supported — only tokens that explicitly implement the standard (primarily **USDC, EURC, AUSD**) can use it natively. Major stablecoins like **USDT and DAI** do not support EIP-3009, which limits x402's token ecosystem. For those tokens, **Permit2** (Uniswap's universal approval contract) serves as a workaround but adds a pre-approval step.[^12][^6][^3]
